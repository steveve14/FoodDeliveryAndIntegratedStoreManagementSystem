# 12. K8s 기반 운영 가이드

기준일: 2026-02-25

## 1. 목표
- 모든 백엔드 서비스를 Kubernetes 기반으로 배포/운영
- 서비스 확장성과 장애 복구 자동화
- 환경(dev/staging/prod) 분리와 표준화된 배포 절차 확보

## 2. 대상 서비스 (현재 디렉토리 기준)
- `service-gateway`
- `service-discovery`
- `service-auth`
- `service-user`
- `service-store`
- `service-order`
- `service-delivery`
- `service-event`

## 3. 권장 클러스터 구성
- Namespace 분리
  - `food-dev`
  - `food-staging`
  - `food-prod`
- 네트워크 접근 정책
  - 외부 진입은 `service-gateway`만 Ingress 공개
  - 나머지 서비스는 `ClusterIP` 내부 통신만 허용
- 구성 관리
  - 일반 설정: `ConfigMap`
  - 민감정보: `Secret`

## 4. 배포 표준

### 4.1 기본 리소스
각 서비스별 최소 리소스:
- `Deployment`
- `Service(ClusterIP)`
- `HorizontalPodAutoscaler` (운영 환경)
- `PodDisruptionBudget` (운영 환경)

### 4.2 프로브 표준
- `readinessProbe`: 트래픽 수신 준비 상태 확인
- `livenessProbe`: 프로세스 생존 상태 확인
- `startupProbe`: 초기 부팅 긴 서비스 보호

### 4.3 리소스 표준(초안)
- requests
  - cpu: `200m`
  - memory: `512Mi`
- limits
  - cpu: `1000m`
  - memory: `1Gi`

## 5. Ingress/도메인 전략
- 단일 진입점: `api.<domain>` -> `service-gateway`
- TLS 필수(운영)
- 인증서: cert-manager + ClusterIssuer 권장

## 6. 데이터/상태 저장 전략
- 애플리케이션은 Stateless 원칙
- DB/Redis/Kafka는 관리형 서비스 또는 별도 Stateful 인프라로 분리
- 백업/복구 정책(RPO/RTO) 문서화 필수

## 7. 관측성(Observability)
- 메트릭: Prometheus + Grafana
- 로그: Loki(or ELK)
- 트레이싱: OpenTelemetry + Tempo/Jaeger
- 필수 알림
  - Pod 재시작 급증
  - 에러율 임계치 초과
  - p95 응답시간 임계치 초과

## 8. CI/CD 권장 흐름
1. PR 머지
2. 이미지 빌드 + 취약점 스캔(Trivy)
3. 이미지 레지스트리 푸시
4. Helm/Kustomize 매니페스트 업데이트
5. Argo CD(또는 Flux)로 GitOps 배포

## 9. 단계별 도입 로드맵

### Phase 1 (기초)
- 서비스별 Dockerfile 정리
- 공통 Helm Chart 또는 Kustomize 베이스 구축
- dev namespace 배포 자동화

### Phase 2 (안정화)
- staging/prod 분리
- HPA/PDB/네트워크정책 적용
- 모니터링/알림 연동

### Phase 3 (고도화)
- 카나리/블루그린 배포
- SLO 기반 자동 스케일링 고도화
- 장애 훈련(Chaos 실험) 도입

## 10. 체크리스트
- [ ] 서비스별 K8s 매니페스트(또는 Helm 값 파일) 작성 완료
- [ ] gateway 외 외부 공개 서비스 없음
- [ ] readiness/liveness/startup probe 적용
- [ ] ConfigMap/Secret 분리 적용
- [ ] 모니터링 대시보드 및 알림 채널 연결
- [ ] 롤백 절차 문서화
