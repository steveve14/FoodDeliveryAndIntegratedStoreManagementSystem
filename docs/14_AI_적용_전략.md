# 14. AI 적용 전략 및 실행 계획

> **작성일**: 2026-03-16  
> **기준 코드/문서**: `service-order`, `service-store`, `service-user`, `service-event`, `web-user`, `docs/10`, `docs/11`, `docs/13`

## 1. 목적

현재 플랫폼은 주문/매장/장바구니/찜/이벤트 데이터가 이미 서비스별로 분리되어 있고, 고객 주문 플로우 E2E도 통과한 상태입니다.  
이 문서는 현행 코드 기준으로 **즉시 적용 가능한 AI 과제**를 정의하고, 단계별 실행 계획(아키텍처/보안/운영 포함)을 제시합니다.

## 2. 현재 상태 진단 (코드 기반)

### 2.1 이미 확보된 AI 입력 데이터

- 주문 이력: `orders`, `order_item` (service-order)
- 장바구니 행동: `cart_items` (service-user)
- 찜 데이터: `favorite_stores` (service-user)
- 이벤트 로그: `events`의 `type`, `payload` (service-event)
- 매장/메뉴 메타데이터: `store`, `menu` + 카테고리 정규화 (service-store)

### 2.2 현재 로직의 한계 (AI 적용 여지)

- 재주문 추천이 규칙 기반(최근 주문 3개)으로 제한됨
- 매장 ETA/태그/프로모션이 카테고리 고정 규칙 위주
- 이벤트 `payload`는 저장되지만 고급 분석/요약 파이프라인이 없음
- 이상 주문, 취소 패턴, 수요 급증 대응 등의 예측 기능 부재

## 3. AI 우선 적용 대상 (Priority)

## P0. 고객 개인화 추천 (2~4주)

- 목표: 홈 화면 추천 정확도와 장바구니 전환율 개선
- 적용 위치:
  - `web-user` 홈/재주문 섹션
  - `service-user`의 찜/장바구니, `service-order`의 주문 이력 기반
- 출력:
  - 사용자별 추천 매장 Top N
  - 사용자별 추천 메뉴 Top N
- 초기 모델:
  - 규칙+통계 하이브리드(최근성, 빈도, 카테고리 선호)
  - 이후 점진적으로 랭킹 모델(XGBoost/LightGBM) 전환

## P1. ETA 예측 고도화 (2~3주)

- 목표: 카테고리 고정 ETA를 실측 기반 ETA로 보정
- 적용 위치:
  - `service-store`의 `resolveEta(...)` 결과를 보정값과 결합
- 입력:
  - 카테고리, 주문 시각(시간대), 주문량, 매장별 과거 주문 처리시간
- 출력:
  - 주문 단위 ETA 예측값(예: 24~31분)

## P1. 운영 이상탐지/알림 (2~4주)

- 목표: 취소 급증, 비정상 상태 전이, 주문 실패율 급증 조기 탐지
- 적용 위치:
  - `service-order`, `service-delivery`, `service-event`
- 출력:
  - 이상 스코어 + 운영자 알림 이벤트

## P2. 사장님/운영자 AI 어시스턴트 (후속)

- 목표: 자연어로 "오늘 취소율 왜 올랐는지", "점심 피크 추천 인력" 질의 지원
- 적용 위치:
  - `web-shop`, `web-admin`
- 방식:
  - RAG(운영지표 + 정책 문서 + 최근 이벤트 요약)

## 4. 권장 아키텍처

### 4.1 서비스 구성

- 신규 `service-ai`(가칭) 추가
  - `recommendation` API
  - `eta-prediction` API
  - `anomaly-score` API
- 기존 서비스 역할 유지
  - 도메인 CRUD/권한/트랜잭션은 기존 `service-*`가 담당
  - AI 서비스는 "예측/추천 결과"만 제공

### 4.2 데이터 파이프라인

1. 단기: DB 배치 추출 + `service-event` 이벤트 적재 활용
2. 중기: `docs/예정/Kafka_도입_계획.md`와 연계해 스트리밍 피처 파이프라인 전환
3. 저장: Postgres 내 `ai_feature`, `ai_prediction_log` 스키마 분리 권장

### 4.3 추론 경로

- 실시간 추론: Gateway -> service-ai (REST)
- 배치 추론: 스케줄러(예: 10분/1시간)로 추천 캐시 갱신
- 캐시: Redis 도입 전에는 DB 캐시 테이블로 시작 가능

## 5. API 초안 (v1)

### 5.1 추천 API

- `GET /api/v1/ai/recommendations/home?userId={id}&limit=10`
- 응답 예시:

```json
{
  "success": true,
  "code": 200,
  "message": "ok",
  "data": {
    "stores": [
      { "storeId": "s1", "score": 0.91, "reason": "최근 30일 재주문 빈도" }
    ],
    "menus": [
      { "menuId": "m1", "score": 0.86, "reason": "찜 카테고리 + 장바구니 패턴" }
    ]
  }
}
```

### 5.2 ETA 예측 API

- `POST /api/v1/ai/eta/predict`
- 요청: `storeId`, `orderCreatedAt`, `itemCount`, `category`, `queueDepth`
- 응답: `etaMin`, `etaMax`, `confidence`

### 5.3 이상탐지 API

- `POST /api/v1/ai/anomaly/orders`
- 요청: 주문/배달 상태 이벤트 집합
- 응답: `anomalyScore`, `anomalyType`, `actionRecommendation`

## 6. 모델 운영(MLOps) 최소 기준

### 6.1 학습/배포

- 모델 버전 규칙: `model_name:major.minor.patch`
- 배포 방식: Shadow -> Canary -> Full
- 롤백 기준: CTR/주문전환/오탐률 임계치 이탈 시 즉시 이전 버전 복귀

### 6.2 관측성 지표

- 온라인 지표:
  - 추천 클릭률(CTR), 주문 전환율(CVR), 평균 주문금액(AOV)
  - ETA MAE, P95 ETA 오차
  - 이상탐지 precision/recall(사후 라벨링)
- 시스템 지표:
  - 추론 p95 latency, timeout rate, fallback rate

## 7. 보안/개인정보/거버넌스

- 개인식별정보(PII) 최소화:
  - 모델 입력에는 해시된 사용자 키 사용
  - 이메일/전화번호 원문은 학습 피처에서 제외
- 토큰/권한:
  - 기존 Gateway JWT 검증 체계 재사용
  - AI API도 동일한 `X-User-Id`, `X-User-Role` 컨텍스트 사용
- 감사 가능성:
  - 추천 결과와 근거(`reason`)를 `ai_prediction_log`에 저장
  - 운영자가 결과 재현 가능하도록 피처 스냅샷 보관

## 8. 단계별 실행 로드맵

### Phase A (주 1~2): 기반 구축

- `service-ai` 스캐폴드 생성
- 추천 API 1차(규칙+통계) 구현
- `web-user` 홈 추천 슬롯 연동
- KPI 대시보드 초안(CTR/CVR) 추가

### Phase B (주 3~4): 정확도 개선

- ETA 예측 1차 모델 추가
- 주문/배달 이상탐지 배치 잡 추가
- A/B 테스트 플래그 적용

### Phase C (주 5~8): 운영 고도화

- Kafka 이벤트 스트림 연결(예정 문서와 연동)
- 온라인 피처 스토어/캐시 고도화
- canary 자동화 + 드리프트 모니터링 적용
