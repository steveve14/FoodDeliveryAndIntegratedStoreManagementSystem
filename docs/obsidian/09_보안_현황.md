## ✅ 완료된 보안 항목 (2026-04-05 기준)

### 1. Gateway 인증 필터
- **상태**: ✅ 완료
- 인증이 필요한 라우트에 `AuthorizationHeaderFilter` 적용 유지

### 2. Refresh Token role 클레임 누락
- **상태**: ✅ 완료
- refresh 경로에서도 role 클레임 유지 → 재발급 Access Token 권한 누락 해소

### 3. DTO 유효성 검증
- **상태**: ✅ 완료
- 요청 DTO에 Bean Validation 적용, 전 서비스 Controller/예외 처리기 정렬

### 4. 에러 핸들러 내부 정보 노출
- **상태**: ✅ 완료
- 클라이언트: 일반화된 오류 메시지만 노출
- 상세 예외: 서버 로그로만 제한

### 5. 헤더 스푸핑 방어 (서비스 내부 JWT 재검증)
- **상태**: ✅ 1차 완료
- 대상: `service-user`, `service-store`, `service-order`, `service-delivery`, `service-event`
- `Authorization` 헤더 또는 `access-token` 쿠키에서 JWT를 직접 검증
- `X-User-Id`, `X-User-Role` 헤더와 토큰 클레임 불일치 시 요청 차단
- **한계**: 내부 서비스 포트가 여전히 외부 노출 → 네트워크 레벨 차단 추가 필요

### 6. IDOR / 민감 엔드포인트 접근제어

| 엔드포인트 | 적용 권한 |
|---|---|
| `GET /api/v1/orders/{id}` | `USER`, `STORE`, `ADMIN` + 소유권/역할 검증 |
| `GET /api/v1/deliveries/{id}` | `STORE`, `ADMIN` |
| `GET /api/v1/orders/frontend/customer-summaries` | `STORE`, `ADMIN` |
| `GET /api/v1/users/me/addresses/**` | `USER`, `ADMIN` |
| `GET /api/v1/users/frontend/**` | `ADMIN` |
| `service-event` 이벤트 생성/조회 | 역할 기반 접근 제어 |

### 7. Auth 쿠키 보안

```
access-token, refresh-token
├── SameSite=Lax (기본)
├── Secure=true  (기본 / prod)
└── Secure=false (local 프로파일만)
```

쿠키 발급/삭제: `ResponseCookie` 방식으로 변경

### 8. Refresh Token 저장 방식

- **이전**: 평문 저장
- **현재**: **SHA-256 해시** 저장 ✅

## 🔐 JWT 구조

```
Access Token
├── 알고리즘: HS384
├── 라이브러리: jjwt 0.12.6 (Jakarta EE 호환)
├── 클레임: sub(userId), role, iat, exp
└── 유효기간: 짧음 (주로 15~30분)

Refresh Token
├── DB 저장: SHA-256 해시값
├── HttpOnly Cookie로만 전달
└── 유효기간: 김 (주로 7~30일)
```
